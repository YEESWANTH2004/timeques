Q 01
Two-pointer scan with early exit on a sorted array
array loop two-pointer 
int twoSum(int[] arr, int target) {   // arr is sorted
    int left = 0, right = arr.length - 1;
    while (left < right) {
        int sum = arr[left] + arr[right];
        if (sum == target)  return 1;
        else if (sum < target) left++;
        else right--;
    }
    return -1;
}
What is the time complexity?
AO(n²)
BO(n)
CO(log n)
DO(n log n)

Even though there are two pointers, they never cross — each step either moves left right or right left. In total, left + right together make at most n moves.

  • At every iteration, one pointer advances → at most n total iterations
  • No nested looping — both pointers share the same while loop
  • Two-pointer on a sorted array is always O(n), not O(n²)

Trick: Two variables inside ONE loop ≠ nested loop. Count how many total steps the pointers together take.

Answer: O(n)
Q 02
Find all pairs in array whose sum equals target (no sorting)
array nested loop 
void findPairs(int[] arr, int target) {
    int n = arr.length;
    for (int i = 0; i < n; i++) {
        for (int j = i + 1; j < n; j++) {
            if (arr[i] + arr[j] == target) {
                System.out.println(arr[i] + " + " + arr[j]);
            }
        }
    }
}
What is the time complexity?
AO(n)
BO(n log n)
CO(n²)
DO(2n)

The inner loop starts at j = i+1, not 0. People often think this halves the work and changes the complexity — it doesn't!

  • When i=0: inner runs n-1 times
  • When i=1: inner runs n-2 times
  • Total = (n-1)+(n-2)+...+1 = n(n-1)/2
  • n(n-1)/2 → drop constant ½ and lower term → O(n²)

Trick: Any double nested loop comparing all pairs, even with j = i+1, is O(n²). The ½ constant gets dropped.

Better approach: Use a HashSet → O(n) time!

Answer: O(n²)
Q 03
Binary search inside a loop over array elements
array loop nested binary search 
int countTargets(int[] arr, int[] queries) {
    int count = 0;
    // arr is sorted, arr.length = n
    // queries.length = m
    for (int q : queries) {           // runs m times
        int l = 0, r = arr.length - 1;
        while (l <= r) {              // binary search: O(log n)
            int mid = (l + r) / 2;
            if (arr[mid] == q) { count++; break; }
            if (arr[mid] < q) l = mid + 1;
            else r = mid - 1;
        }
    }
    return count;
}
What is the time complexity? (n = arr size, m = queries size)
AO(n log n)
BO(m + n)
CO(m · n)
DO(m log n)

Two distinct sizes appear: m (queries) and n (array). Outer loop runs m times, inner binary search runs log n times.

  • Outer loop: m iterations (one per query)
  • Inner while: O(log n) — standard binary search on array of size n
  • Nested → multiply: m × log n = O(m log n)

Trick: When two different input sizes exist, keep both in the answer. Don't collapse m and n into a single variable unless they're equal.

Answer: O(m log n)
Q 04
Prefix sum build + range query
array loop prefix sum 
int[] prefix;

void build(int[] arr) {          // called once
    prefix = new int[arr.length + 1];
    for (int i = 0; i < arr.length; i++)
        prefix[i+1] = prefix[i] + arr[i];  // O(n)
}

int query(int l, int r) {         // called q times
    return prefix[r+1] - prefix[l]; // O(1) each
}
// Total: build once, then q queries
What is the TOTAL time complexity for build + q queries?
AO(n + q)
BO(n · q)
CO(n²)
DO(q log n)

The whole point of prefix sums is to trade build time for O(1) queries.

  • build() runs one loop of n steps → O(n)
  • query() is a single subtraction → O(1) per call
  • q queries × O(1) each = O(q)
  • Total (sequential): O(n) + O(q) = O(n + q)

Contrast with naive approach: Without prefix sums, each range query would loop through the subarray → O(n) per query → O(nq) total. Prefix sums are a massive win!

Answer: O(n + q)
Q 05
Triple nested loop — counting triplets
array nested loop 
int countTriplets(int[] arr) {
    int n = arr.length, count = 0;
    for (int i = 0; i < n; i++) {
        for (int j = i + 1; j < n; j++) {
            for (int k = j + 1; k < n; k++) {
                if (arr[i]+arr[j]+arr[k] == 0)
                    count++;
            }
        }
    }
    return count;
}
What is the time complexity?
AO(n²)
BO(n³)
CO(n! )
DO(3n)

Three nested loops, each bounded by n (even though they start at i+1, j+1 etc.).

  • Total combinations = C(n,3) = n(n-1)(n-2)/6
  • Drop constants and lower-order terms: O(n³)
  • Starting at i+1, j+1 only halves/thirds the work — constants that disappear in Big O

Rule: k nested loops each bounded by n → O(nᵏ). Three loops = O(n³), regardless of whether inner bounds shift.

Better: Sort array + two-pointer for each i → O(n²). Or use a HashSet → O(n²).

Answer: O(n³)
Q 06
Loop with logarithmic inner shrink — matrix row search
array loop logarithmic inner 
void process(int[][] matrix, int n) {
    // matrix is n×n
    for (int row = 0; row < n; row++) {     // n rows
        int lo = 0, hi = n - 1;
        while (lo <= hi) {                    // binary search on row
            int mid = (lo + hi) / 2;
            if (matrix[row][mid] == 0) { hi = mid - 1; }
            else                           { lo = mid + 1; }
        }
        System.out.println("zeros in row: " + lo);
    }
}
What is the time complexity?
AO(n²)
BO(n)
CO(n log n)
DO(log n)

Outer loop runs over n rows. Inside each row, a binary search runs on n columns.

  • Outer for-loop: n iterations
  • Inner while (binary search on row of size n): O(log n)
  • Nested → multiply: n × log n = O(n log n)

Contrast with O(n²): If the inner loop were a linear scan (lo++ each time), it would be O(n²). Binary search cuts it to O(n log n).

Trick: Outer O(n) × inner O(log n) = O(n log n). Same pattern as Merge Sort!

Answer: O(n log n)
Q 07
Sliding window — max subarray sum of size k
array loop sliding window 
int maxSubarraySum(int[] arr, int k) {
    int n = arr.length;
    int windowSum = 0, maxSum = 0;

    for (int i = 0; i < k; i++)         // first window: O(k)
        windowSum += arr[i];
    maxSum = windowSum;

    for (int i = k; i < n; i++) {        // slide: O(n-k)
        windowSum += arr[i] - arr[i - k];
        maxSum = Math.max(maxSum, windowSum);
    }
    return maxSum;
}
What is the time complexity?
AO(n)
BO(n · k)
CO(k)
DO(n²)

Classic sliding window — two sequential loops, neither nested.

  • First loop builds initial window: O(k)
  • Second loop slides window across remaining elements: O(n - k)
  • Sequential: O(k) + O(n-k) = O(n)
  • Each element is added and removed from window exactly once

Why not O(n·k)? A naive approach would recompute the sum from scratch for each window (a nested loop) → O(n·k). Sliding window avoids this by just adding one element and removing one → O(1) per slide → O(n) total.

Trick: Sliding window converts O(n·k) naive → O(n) by reusing previous computation.

Answer: O(n)
Q 08
Loop with inner loop shrinking by half each outer iteration
array loop shrinking window 
void mystery(int[] arr) {
    int n = arr.length;
    for (int i = 1; i < n; i++) {          // outer: n times
        for (int j = 0; j < n; j += i) {   // inner: n/i times
            System.out.print(arr[j] + " ");
        }
        System.out.println();
    }
}
What is the time complexity?
AO(n²)
BO(n)
CO(n log n)
DO(n√n)

The inner loop step is j += i, so it runs n/i times for each outer value of i.

  • i=1: inner runs n/1 = n times
  • i=2: inner runs n/2 times
  • i=3: inner runs n/3 times
  • Total = n/1 + n/2 + n/3 + ... + n/n = n × (1 + 1/2 + 1/3 + ... + 1/n)
  • The harmonic series: (1 + 1/2 + ... + 1/n) ≈ ln(n) ≈ log n
  • Total = n × log n = O(n log n)

This is the Harmonic Series pattern! Whenever inner loop runs n/i iterations summed over i=1..n, the total is O(n log n). This appears in the Sieve of Eratosthenes too.

Answer: O(n log n)
Q 09
Merge two sorted arrays then binary search
array loop merge + search 
boolean mergeAndSearch(int[] a, int[] b, int target) {
    // a.length = n,  b.length = m
    int[] merged = new int[a.length + b.length]; // size n+m
    int i = 0, j = 0, k = 0;

    while (i < a.length && j < b.length) {  // merge: O(n+m)
        if (a[i] <= b[j]) merged[k++] = a[i++];
        else              merged[k++] = b[j++];
    }
    while (i < a.length) merged[k++] = a[i++];
    while (j < b.length) merged[k++] = b[j++];

    int lo = 0, hi = merged.length - 1;  // binary search: O(log(n+m))
    while (lo <= hi) {
        int mid = (lo + hi) / 2;
        if (merged[mid] == target) return true;
        if (merged[mid] < target)  lo = mid + 1;
        else                       hi = mid - 1;
    }
    return false;
}
What is the TOTAL time complexity?
AO(log(n + m))
BO(n + m)
CO((n + m) log(n + m))
DO(n · m)

Two sequential phases — add their complexities, then keep the dominant term.

  • Merge phase: each element from a and b is touched once → O(n + m)
  • Binary search on merged array of size (n+m): O(log(n+m))
  • Total: O(n+m) + O(log(n+m))
  • Which dominates? O(n+m) grows much faster than O(log(n+m))
  • Keep dominant term: O(n + m)

Key insight: The merge step dominates the binary search step. Even though binary search sounds "complex," it's actually very fast — the linear merge is the bottleneck.

Answer: O(n + m)
Q 10
Recursive array processing — divide in thirds
array recursion loop 
void solve(int[] arr, int lo, int hi) {
    if (hi - lo < 1) return;          // base case

    int third = (hi - lo) / 3;

    // Do O(n) work at this level
    for (int i = lo; i <= hi; i++)
        arr[i] = arr[i] * 2;

    // Recurse on TWO thirds
    solve(arr, lo,          lo + third);
    solve(arr, lo + third,  lo + 2*third);
}
What is the time complexity?
AO(n²)
BO(n log₃ n)
CO(n)
DO(n log n)

Apply the Master Theorem: T(n) = 2·T(n/3) + O(n).

  • Problem splits into 2 sub-problems each of size n/3
  • Work done at each level: O(n) (the for loop)
  • Number of recursion levels: log₃(n)
  • Total work per level: varies, but dominated by top level = O(n)
  • Master Theorem case: a=2, b=3, d=1. Since log₃(2) < 1: O(n log n) applies… wait
  • More precisely: nlog₃2 ≈ n^0.63 < n¹, so work dominated by leaves? No — the O(n) work per level dominates → each level does O(n) total → log₃n levels → O(n log n)

Intuition: Each recursion level processes all n elements (across different sub-calls). There are log₃(n) levels. So total = n × log₃(n). Since log₃(n) = log(n)/log(3), and log(3) is a constant, this is simply O(n log n).

Trick: Whenever a recursion does O(n) work per level and has O(log n) levels (regardless of base 2, 3, etc.) → total is O(n log n).

Answer: O(n log n)
